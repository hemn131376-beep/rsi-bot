# بوت RSI الحلال على تيليجرام

بوت Node.js يراقب مؤشّر **RSI الحيّ** على مدار الساعة لقائمة حلال مغلقة (687 رمزاً) عبر
**Binance → OKX → KuCoin → MEXC**، ويرسل تنبيهات تيليجرام عند دخول مناطق التشبّع البيعي،
مع رسالة مثبّتة حيّة وملخّص (digest) كلّ 15 دقيقة.

> صُمِّم ليعمل خدمةً دائمة على **Railway + Postgres**. لا يحتاج أيّ مفاتيح API للمنصّات
> (يستخدم بيانات السوق العامّة فقط).

---

## ما الذي يفعله البوت

- **القائمة المغلقة:** يراقب فقط الرموز الموجودة في `data/halal_whitelist.csv`. لا شيء خارجها.
- **توزيع المنصّات:** كل رمز يُسنَد لأوّل منصّة (بترتيب التدرّج) تدرجه فعلياً بزوج **سائل**،
  بأولوية الأزواج `USDT → USDC → BTC`. ما لا تتوفّر له سيولة كافية أو غير مُدرَج يدخل قائمة «غير مراقَب».
- **الأطر الزمنية:** ‎1H / 4H / 1D / 1W.
- **RSI:** فترة 14 بتنعيم Wilder، محسوب **لحظياً على الشمعة المفتوحة** (لا ننتظر الإغلاق).
- **منطق الدخول/الخروج (hysteresis):**
  - دخول فوري عندما ‎`provRSI < 7`.
  - يبقى ضمن المنطقة طالما ‎`< 10`.
  - خروج عندما ‎`> 10`. الفجوة 7↔10 تمنع الرفرفة.
- **الرسالة المثبّتة:** رسالة حيّة مثبّتة تُحدَّث دورياً (مع حدّ لتفادي الحظر 429).
- **الملخّص كلّ 15 دقيقة:** مُحاذى لـ ‎`:00/:15/:30/:45` مع أسهم اتجاه (↓ يتعمّق / ↑ يتعافى).
- **الموثوقية:** عند أيّ انقطاع يُعاد بذر حالة Wilder من REST قبل استئناف WebSocket
  (لا نثق بحالة قديمة)، مع watchdog و keepalive و backoff أُسّي.

## المتطلّبات

- Node.js **20+**
- قاعدة Postgres (يوفّرها Railway بنقرة)
- بوت تيليجرام (توكن من BotFather) ومعرّف الدردشة/القناة

---

## الإعداد المحلّي

```bash
npm install
cp .env.example .env      # ثم عبّئ القيم
npm test                  # اختبار منطقي بلا شبكة (يجب أن تنجح كل الاختبارات)
npm start                 # تشغيل فعلي (يحتاج اتصالاً بالمنصّات وتيليجرام)
```

للتجربة السريعة بلا Postgres:

```bash
NO_DB=1 npm start
```

> في وضع `NO_DB` تُحفظ الحالة في الذاكرة فقط ولا تنجو من إعادة التشغيل. للإنتاج استخدم Postgres.

## متغيّرات البيئة

| المتغيّر | إلزامي | الوصف |
|---|---|---|
| `TELEGRAM_BOT_TOKEN` | ✅ | توكن البوت من BotFather |
| `TELEGRAM_CHAT_ID` | ✅ | معرّف الدردشة أو القناة لإرسال التنبيهات |
| `DATABASE_URL` | ✅ | رابط اتصال Postgres |
| `WHITELIST_PATH` | ✗ | مسار ملف القائمة (افتراضي `data/halal_whitelist.csv`) |
| `MIN_VOLUME_USD` | ✗ | الحدّ الأدنى لحجم 24س بالدولار، فلتر السيولة (افتراضي `75000`) |
| `SEED_DELAY_MS` | ✗ | الفاصل بين طلبات بذر REST بالميلي ثانية (افتراضي `150`؛ ارفعه لو واجهت حظراً) |
| `NO_DB` | ✗ | `1` لتشغيل تجريبي بلا Postgres |
| `DEBUG` | ✗ | `1` لسجلّات تفصيلية |

## الحصول على التوكن ومعرّف الدردشة

1. **التوكن:** راسِل [@BotFather](https://t.me/BotFather) على تيليجرام → `/newbot` → اتبع الخطوات → انسخ التوكن.
2. **معرّف الدردشة:**
   - للدردشة الخاصة: راسِل [@userinfobot](https://t.me/userinfobot) ليعطيك `id`.
   - للقناة/المجموعة: أضِف البوت مشرفاً، ثمّ استخدم معرّف القناة (يبدأ غالباً بـ `-100…`).
   - تأكّد أنّ البوت يملك صلاحية **إرسال الرسائل** و**تثبيتها** في القناة.

## استبدال القائمة بالحقيقية (687 رمز)

الملف المرفق `data/halal_whitelist.sample.csv` عيّنة من 20 رمزاً للتجربة فقط.
**استبدِله** بملفّك الحقيقي بنفس الاسم `data/halal_whitelist.csv`:

- يكفي عمود واحد اسمه `base` (رموز الأصل: `BTC`, `ETH`, …).
- تُقبَل أيضاً الأعمدة `symbol`/`asset`/`coin`/`ticker`، وتُنظَّف لاحقة الزوج (`USDT`/`USDC`/`BTC`) تلقائياً.
- الأسطر التي تبدأ بـ `#` تُتجاهَل.

---

## النشر على Railway

1. ارفع المشروع إلى مستودع GitHub (لا ترفع `node_modules` ولا `.env`).
2. على Railway: **New Project → Deploy from GitHub repo**.
3. أضِف خدمة **Postgres** من زرّ *New* داخل المشروع (يُولّد `DATABASE_URL` تلقائياً، اربطه بالخدمة).
4. في **Variables** أضِف: `TELEGRAM_BOT_TOKEN` و`TELEGRAM_CHAT_ID` (و`DATABASE_URL` إن لم يُربط تلقائياً).
5. النشر يعمل تلقائياً عبر `Procfile`/`railway.json` بأمر `node src/index.js`.
6. أعِد التشغيل تلقائي عند الفشل (مُهيّأ في `railway.json`).

> الجداول تُنشأ تلقائياً عند الإقلاع (`monitored_zone`, `bot_state`, `coin_routing`).

## أوامر تيليجرام

- `/list` — يعرض الأصول داخل منطقة التشبّع البيعي حالياً (مرتّبة حسب RSI).
- `/status` — حالة البوت: عدد المراقَب/غير المراقَب، توزيع المنصّات، آخر إعادة اتصال، ووقت التشغيل.

---

## ملاحظة مهمّة قبل الإنتاج

نقاط نهاية WebSocket/REST وحدودها على المنصّات الأربع قد **تتغيّر مع الوقت**.
المنطق الأساسي (RSI، آلة الحالة، التوجيه، التقسيم) مُختبَر offline، لكن يُنصَح بالتحقّق من
نقاط الوصول الحيّة لكل منصّة قبل الاعتماد عليه إنتاجياً، وضبط `SEED_DELAY_MS` و`MIN_VOLUME_USD`
حسب حاجتك.

## البنية باختصار

```
src/
  index.js        ← المنسّق الرئيسي وتسلسل الإقلاع
  config.js       ← كل الإعدادات والعتبات
  whitelist.js    ← تحميل القائمة المغلقة وتنظيف الرموز
  router.js       ← توزيع الرموز على المنصّات (تدرّج + سيولة + أولوية الزوج)
  rsi.js          ← مؤشّر Wilder RSI (seed/provisional/commit)
  engine.js       ← حالة RSI لكل (منصّة|رمز|إطار)
  stateMachine.js ← منطق الدخول/البقاء/الخروج + الملخّص
  scheduler.js    ← مؤقّتات الملخّص والتحديث والمراقبة
  db.js           ← Postgres (مع وضع NO_DB في الذاكرة)
  telegram.js     ← طبقة تيليجرام (تنبيهات/مثبّتة/ملخّص/أوامر)
  format.js       ← دوال تنسيق نقيّة قابلة للاختبار
  exchanges/      ← محوّلات Binance/OKX/KuCoin/MEXC + القاعدة المشتركة
test/
  integration.test.js ← اختبار منطقي شامل بلا شبكة
```
